۱. پروژه چیه و چرا این سناریو؟

ما یک سامانه سنجش شناختی داریم. کاربر (شهروند) وارد می‌شود، لیست آزمون‌های قابل انجام را می‌بیند، می‌تواند جستجو کند و لیست را مرتب کند (بر اساس عنوان، سطح، زمان و…).

برای درس تست نرم‌افزار، نمی‌شود کل سیستم Django + React را یک‌جا تست کرد؛ باید یک واحد تحت آزمون (UUT) انتخاب کنیم. ما تابع process_catalog در catalog.py را UUT اصلی گرفتیم، به‌علاوه چهار الگوریتم جدا: bubble_sort، merge_sort، linear_search، binary_search.

این تابع در بک‌اند صدا زده می‌شود: لیست آزمون‌ها از دیتابیس می‌آید، داخل process_catalog اول (در صورت نیاز) فیلتر می‌شود، بعد مرتب، و خروجی به React برمی‌گردد.

۲. الگوریتم‌ها توی سیستم واقعی چیکار می‌کنند؟

الگوریتمکارکی استفاده می‌شود؟
linear_search از اول لیست می‌گردد؛ اگر کلمه‌ای که کاربر تایپ کرده داخل title یا description باشد، آن آزمون را نگه می‌دارد همیشه — در catalog_bridge.py جستجو همیشه linear است
binary_search روی لیست مرتب، تطبیق دقیق عنوان را پیدا می‌کند فقط در تست‌ها (ACOC/CFG) — کاربر در UI انتخابش نمی‌کند
bubble_sort مرتب‌سازی با مقایسه جفت‌های مجاور اگر تعداد آزمون ≤ ۱۲ تا
merge_sort تقسیم لیست، مرتب کردن نیمه‌ها، ادغام اگر تعداد آزمون > ۱۲ تا
process_catalog همه را به هم وصل می‌کند + meta برمی‌گرداند (چند تا بود، چند تا ماند، کدام الگوریتم) هر بار که API لیست آزمون را فیلتر/مرتب می‌کند

کاربر فقط «جستجو» و «مرتب بر اساس …» می‌بیند. انتخاب bubble یا merge پشت صحنه در resolve_algorithms انجام می‌شود (آستانه ۱۲).

۳. روش کلی کار — از کد تا تست

طبق فصل‌های کتاب، این ترتیب را رفتیم:

  1. بخشی از کد را انتخاب کردیم (UUT)
  2. فضای ورودی را افراز کردیم (ISP — Input Space Partitioning)
  3. با ACOC همه ترکیب پارامترها را تست کردیم (فصل ۶)
  4. کد را به CFG تبدیل کردیم و Node / Edge / Prime Path زدیم (فصل ۷)
  5. جهش AOR ساختیم و Mutation Score حساب کردیم (فصل ۹)

نمی‌شود «همه ورودی‌های ممکن» را تست کرد — مثلاً هر رشته جستجو، هر اندازه لیست. برای همین اول فضای ورودی را به کلاس‌های معادل تقسیم می‌کنیم، بعد روی همان کلاس‌ها تست می‌نویسیم.

۵. CFG — گراف و تست‌های Node / Edge / Prime

کد هر تابع را به گراف جریان کنترل تبدیل کردیم: مستطیل = دستور، لوزی = شرط، فلش = جهت اجرا. گراف‌ها در پوشه cfg_graphs/ هستند (مثلاً bubble_sort.html، process_catalog.html).

سه نوع پوشش (فصل ۷)

  • Node Coverage: هر گره (هر بلوک کد/شرط) حداقل یک‌بار اجرا شود.
    مثال: test_bubble_empty_list → گره «if n<=1» با T
  • Edge Coverage: هر یال (هر شاخه T و F) حداقل یک‌بار طی شود.
    مثال: test_bubble_swap_branch_taken → شاخه swap=T
  • Prime Path: مسیر ساده‌ای که زیرمسیر تکراری ندارد — معمولاً کل حلقه‌ها.
    مثال: test_bubble_full_outer_inner_loop

شماره نودها

مثلاً bubble_sort: BS1 تا BS10 — هر شماره یک بلوک در sorting.py. process_catalog: PC1–PC4 و داخل _apply_search: AS1–AS6.

۶. Mutation Testing — ۱۴ جهش، Score چطور حساب شد؟

جهش یعنی یک تغییر کوچک عمدی در کد (مثلاً n-i-1 را بکنی n+i-1). اگر تست‌های ما این باگ را بگیرند → جهش KILLED. اگر تست رد نشود ولی رفتار عوض شده → LIVE. اگر جهش در عمل همان خروجی را بدهد → EQUIVALENT.

نوع جهش ما: AOR (Arithmetic Operator Replacement) — فقط یک عملگر عوض می‌شود.

۹. نکته درباره process_catalog (سؤال‌های معمول)

هر سه مسیر جستجو هم‌نوع برمی‌گردانند؟ بله — همه list[dict]. linear_search اندیس می‌دهد ولی داخل تابع به آیتم تبدیل می‌شود: [items[i] for i in indices].

چرا binary دو بار مرتب می‌کند؟ یک‌بار bubble برای پیش‌شرط جستجوی دودویی، یک‌بار _apply_sort برای ترتیب نهایی کاربر. در production فقط linear استفاده می‌شود.

meta از کجا می‌آید؟ از پارامترهای ورودی بعد از اعتبارسنجی (sort_algo، search_algo)، نه از خروجی _apply_search.
